What this is & how to use it

A shared board for running a Test–Learn–Grow practice in the open: post experiments with a hypothesis and a success measure written before the test starts, run sessions people can sign up to, set challenges others can respond to — then carry the findings that worked forward to a deliberate scale decision.

A note on names: this guide uses the default labels — Experiment, Session, Challenge, Member, points. Your board may rename any of these; the behaviour is the same whatever they are called.

What the board is for

The board runs a Test–Learn–Grow loop and keeps it visible:

  1. Test — make a clear prediction, try something small, measure the result.
  2. Learn — share what you found, honestly, whether it worked or not.
  3. Grow — scale the findings that held up; park the ones that didn't.

Its job is to hold a few useful gates in place — a test must state how you'll know it worked before it starts, and a decision to scale must record the evidence behind it. Everything else is kept light on purpose.

The three activity types

Everything on the board is one of three kinds of activity. You don't have to use all three — an admin can switch Sessions or Challenges off.

Activity types at a glance
Type What it is Lifecycle in short
Experiment A test of a change, with a hypothesis and a success measure. The heart of the board. Designing → Running → Wrapping up → Finding shared → Growing → Scaled
Session A scheduled event others can sign up to — a workshop, talk, or working session. Scheduled → Held → Output shared
Challenge An open prompt or problem inviting responses from others. Open → responses come in → Closed

The experiment lifecycle

An experiment is designed to make you think before you act and reflect after.

  1. Designing. Write the test before running it: a hypothesis ("If we do X, we expect Y because Z"), a predicted outcome, and — required — a success measure plus a baseline. You can't advance without a success measure; that gate is the point.
  2. Running. The test is live.
  3. Wrapping up. Record the measured result against your success measure, and a verdict — did the prediction hold? A validated verdict requires a measured result.
  4. Finding shared. The learning is published for everyone. This is a milestone — the team earns points here.
  5. Growing. Decide what to do with a finding that worked: scale, adopt, re-run, or stop. A scale/adopt decision records its active ingredients, an owner, the evidence strength, and scale readiness. Only scale/adopt earn the grow reward — parking a dead end is honest, not points-worthy.
  6. Scaled. A later review records whether the finding actually held at scale, not just that someone intended to scale it.

Off-ramps exist too: an experiment can be parked, and learnings can spawn a follow-on experiment that links back to its parent. Experiments can also ladder up to a mission/outcome so their impact is attributable. The Pipeline page groups all experiments by stage so you can see where work is stuck.

Using it as a team member

  • Sign in with your organisation account. On a local or demo board you'll get a mock account picker instead.
  • Browse the board. The home page shows your next steps, fresh learning, and the full board. Open any item for detail.
  • Start something from the create pages. For an experiment, you'll be asked for the hypothesis and success measure up front — that's deliberate.
  • Take part without owning. On items you don't own you can still: sign up for or leave a session, join or leave an experiment's team, post an update you authored, and respond to a challenge. These are additive — you add or remove yourself and append your own contributions, but you can't edit other people's content.
  • Keep your profile current. Your Member card lists your expertise, what you'll mentor on, and your stretch goals, so people can find who can help.
  • Watch the leaderboard if points are enabled.

How points work

Points (if enabled) reward finishing and sharing, not just starting. Defaults:

Default points awards
Action Points Who earns
Experiment finding shared100the experiment team
Growing (scale/adopt decision)100the experiment team
Hosting a session (output shared)75the host
Attending a session (output shared)25each attendee
Posting a challenge25the poster

The server awards points, once: you never edit the leaderboard directly, and the same milestone can't pay out twice. Amounts, rank names, and thresholds are all configurable, and points can be turned off entirely.

Roles and permissions

There are no complicated roles — just a sensible trust model, enforced by the server (the interface only hides buttons you can't use; the server is the real gate).

  • Anyone signed in can create activities, take part additively, and edit their own Member card.
  • An item's owner — the poster, a session's host, or a team member — can edit and delete that item fully. Who created it, when, and what type can never be changed, even by the owner.
  • Admins can edit any item, correct the leaderboard, and use the Admin page to set branding, terminology, features, points, and the colour theme. Bootstrap admins can't be removed from the interface, so a board can't lock everyone out.

For admins: configuring a board

The Admin page edits the live config. The notable guardrail: theme colours are validated to WCAG 2.2 AAA contrast on save — the board refuses a palette whose text falls below 7:1, so you can't accidentally ship an inaccessible theme. You can change branding, terminology, which features are enabled, points rules, the colour theme, and the skills catalogue that populates Member cards.
